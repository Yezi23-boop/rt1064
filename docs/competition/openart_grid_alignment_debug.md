# OpenART 16x12 网格对齐调试方法

本文记录 OpenART #1 做 16x12 地图识别时，四角标定后采样圆点仍然对不上格子中心的排查方法。适用对象主要是：

```text
openmv/openart_plus_grid_recognizer_pure.py
openmv/openart_plus_grid_recognizer.py
```

## 先明确坐标含义

`MAP_CORNERS` 必须填写完整 16x12 地图外边界的四个角，顺序固定为：

```text
左上 -> 右上 -> 右下 -> 左下
```

这里的“外边界”不是屏幕黑框、游戏窗口标题栏，也不是最外圈墙体格子的中心。如果把四个角标在墙体格子中心，后续 `(col + 0.5) / 16` 和 `(row + 0.5) / 12` 算出来的采样点会整体错半格。

当前地图比例是 16:12，也就是 4:3，和 QVGA 320x240 / VGA 640x480 一致。因此在四角正确且镜头无明显畸变时，`rotation_corr(corners=...)` 拉正后可以直接按整张图均分 16 列、12 行。

## 四角标定后仍对不齐的常见原因

1. 四个角标错了，标到了屏幕边框、窗口边框、墙体中心或局部墙体纹理上。
2. 只画 192 个大圆点，肉眼无法判断是圆心不准还是圆半径盖住了边界。
3. 整体有固定偏移或缩放误差，需要对有效采样区域加边距微调。
4. 摄像头或屏幕存在镜头畸变，四角对了但中间或边缘逐渐偏。
5. 实际画面不是严格平面，或者屏幕与摄像头支架有轻微歪斜、松动。
6. 显示路径和识别路径不是同一个坐标系，例如显示原图但识别走拉正图，看到的调试点会让人误解。

## 推荐排查顺序

### 1. 先画完整网格线

只看 192 个圆点不够。调试时应增加一个开关，例如：

```python
DEBUG_DRAW_GRID_LINES = True
DEBUG_POINT_RADIUS = 1
```

在拉正图上画 17 条竖线和 13 条横线。判断标准：

- 网格线能贴住真实格子边界，圆点才有意义。
- 如果原始画面本身是斜方框或梯形，先不要把问题归因到整体平移/缩放；应优先确认四角标定和透视矫正。
- 只有在拉正后的地图已经是规则矩形、但整张网格仍统一偏一点或大一点/小一点时，才考虑边距微调。
- 如果网格线四角贴住但中间弯或边缘偏，考虑镜头畸变或多点标定。

### 2. 区分三类误差

整体偏移或整体缩放：

```text
所有点都偏左、偏右、偏上、偏下，或者整体稍大/稍小。
```

处理方式：仅在透视拉正已经基本成功后，加有效区域边距微调。对于原始画面明显斜视的场景，这不是第一排查项。

透视或角点错误：

```text
某一边越来越偏，或者只有一个角附近严重偏。
```

处理方式：重新标四角，确认四点顺序和地图外边界。

镜头畸变：

```text
四角基本对，但中间或边缘呈弧形偏差。
```

处理方式：尝试 `lens_corr()`，或升级为多点标定。

### 3. 加四个边距参数

当误差主要是整体偏移或缩放时，不要反复乱改四角。更稳定的方式是在拉正后的图上增加有效采样区：

注意：`GRID_*_MARGIN` 只能修正拉正后规则矩形内的小范围统一偏差，不能替代四角透视矫正。若原始画面仍是斜方框、拉正后外框也没有对正，应先修 `MAP_CORNERS` 或透视流程，而不是先调边距。

```python
GRID_LEFT_MARGIN = 0
GRID_RIGHT_MARGIN = 0
GRID_TOP_MARGIN = 0
GRID_BOTTOM_MARGIN = 0
```

采样中心从下面这个区域里均分：

```text
left = GRID_LEFT_MARGIN
right = IMG_WIDTH - GRID_RIGHT_MARGIN
top = GRID_TOP_MARGIN
bottom = IMG_HEIGHT - GRID_BOTTOM_MARGIN
```

然后：

```python
x = left + (col + 0.5) * (right - left) / GRID_COLS
y = top + (row + 0.5) * (bottom - top) / GRID_ROWS
```

这比直接改 `MAP_CORNERS` 更适合修正小范围整体偏差。

### 4. 比较两条识别路径

当前脚本可以拆成两个独立开关：

```python
SHOW_RECTIFIED_VIEW = True
USE_RECTIFIED_RECOGNITION = True
```

含义：

- `SHOW_RECTIFIED_VIEW` 只决定 OpenMV IDE 看到的是原图还是拉正图。
- `USE_RECTIFIED_RECOGNITION` 只决定取色识别是在原图投影点上做，还是在拉正图规则网格上做。

调试时优先使用：

```python
SHOW_RECTIFIED_VIEW = True
USE_RECTIFIED_RECOGNITION = True
```

这样看到的图、画的网格、实际取色都在同一个坐标系里。等标定稳定后，再评估是否切回原图投影点识别以提升速度。

### 5. 必要时做镜头畸变修正

OpenMV 官方提供 `Image.lens_corr()` 用于镜头畸变校正，`Image.rotation_corr(corners=...)` 用于四角透视校正，`Image.find_rects()` 可用于查找矩形候选区域。

参考文档：

- <https://docs.openmv.io/library/omv.image.Image.html>

如果观察到直线边界在画面中变弯，可以尝试流程：

```text
sensor.snapshot()
-> img.lens_corr(...)
-> img.rotation_corr(corners=MAP_CORNERS)
-> 16x12 均分采样
```

注意：`lens_corr()` 会增加耗时，不应在没有明显畸变证据时默认开启。

### 6. 四角仍不够时用多点标定

如果四角、边距、镜头校正后仍然对不齐，说明四点透视模型不足。可升级为多点标定：

```text
上边界标多个点
下边界标多个点
左边界标多个点
右边界标多个点
```

然后对每一行、每一列单独插值出格子中心。这个方法比四角透视更抗畸变和非理想安装，但代码复杂度更高，建议作为最后方案。

多点标定的输入点不是格子中心，而是地图外边界上的网格线交点。建议先按每条边 5 个点设计：

```python
TOP_EDGE_POINTS = (
    (x_col0, y_col0),    # 上边界第 0 列位置，左上角
    (x_col4, y_col4),    # 上边界第 4 列位置
    (x_col8, y_col8),    # 上边界第 8 列位置，中点附近
    (x_col12, y_col12),  # 上边界第 12 列位置
    (x_col16, y_col16),  # 上边界第 16 列位置，右上角
)

BOTTOM_EDGE_POINTS = (
    (x_col0, y_col0),    # 下边界第 0 列位置，左下角
    (x_col4, y_col4),
    (x_col8, y_col8),
    (x_col12, y_col12),
    (x_col16, y_col16),  # 下边界第 16 列位置，右下角
)

LEFT_EDGE_POINTS = (
    (x_row0, y_row0),    # 左边界第 0 行位置，左上角
    (x_row3, y_row3),    # 左边界第 3 行位置
    (x_row6, y_row6),    # 左边界第 6 行位置，中点附近
    (x_row9, y_row9),    # 左边界第 9 行位置
    (x_row12, y_row12),  # 左边界第 12 行位置，左下角
)

RIGHT_EDGE_POINTS = (
    (x_row0, y_row0),    # 右边界第 0 行位置，右上角
    (x_row3, y_row3),
    (x_row6, y_row6),
    (x_row9, y_row9),
    (x_row12, y_row12),  # 右边界第 12 行位置，右下角
)
```

这里的 `x_col4, y_col4` 不是要照抄的变量名，而是 OpenMV 图像中的实际像素坐标。含义是：在地图上边界第 4 列网格线与上边界的交点处，读取并填入这个点的 `(x, y)`。

16 列地图有 17 条竖边界线，编号为 `0..16`；12 行地图有 13 条横边界线，编号为 `0..12`。所以上下边界可先标 `0, 4, 8, 12, 16`，左右边界可先标 `0, 3, 6, 9, 12`。

多点标定生成采样中心的思路：

```text
1. 用左/右边界点插值得到当前行的左端点和右端点。
2. 在这一行的左端点与右端点之间均分 16 列。
3. 每个格子的中心使用 (col + 0.5) / 16 计算。
4. 需要更稳时，再把上/下边界插值得到的结果与左/右插值结果做平均。
```

实现顺序先固定为两阶段：

```text
第一版：只实现 LEFT_EDGE_POINTS + RIGHT_EDGE_POINTS。
第二版：再加入 TOP_EDGE_POINTS + BOTTOM_EDGE_POINTS，做双向平均。
```

第一版的目标是先把“每一行单独拉直并均分”跑通，适合处理每行宽度、斜率随透视变化的问题。它标点少、代码简单、便于验证，适合作为后续 agent 的第一步实现。

第二版再让上/下边界参与，把左右插值得到的点 `P_lr` 与上下插值得到的点 `P_tb` 取平均：

```text
final_point = (P_lr + P_tb) / 2
```

第二版用于处理列方向漂移、边缘畸变或轻微局部弯曲。不要一开始就实现双向平均，避免在标定点还没验证前引入更多变量。

## 当前建议落地顺序

1. 先加 `DEBUG_DRAW_GRID_LINES` 和 `DEBUG_POINT_RADIUS`。
2. 在拉正图上确认网格线是否贴住真实 16x12 格子。
3. 如果整体偏移，加入四个 `GRID_*_MARGIN` 参数。
4. 如果四角对但中间弯，再尝试 `lens_corr()`。
5. 如果仍然不稳定，再考虑多点标定，不要一开始就做自动 ROI 搜索。

## 判断是否合格

调试图里应同时满足：

1. 绿色外框贴住完整地图外边界。
2. 17 条竖线和 13 条横线基本贴住格子边界。
3. 192 个小圆点位于格子内部中心附近，不压到墙体边界或色块边缘。
4. 关闭调试圆点后，字符地图在连续多帧内稳定，不出现墙/空地大面积跳变。
