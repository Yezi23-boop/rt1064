# 掉电保存菜单框架

本文定义 RT1064 Push Box 项目的屏幕菜单与掉电保存边界。目标是在 IPS200 屏幕和 `KEY_1`~`KEY_4` 四按键上提供一个可执行、可选图、可选模式、可掉电恢复的菜单系统。

## 总体决策

- 开机进入多层菜单的 `Home` 主页面。
- `Run` 是 `Home` 下的执行子页面，不替代 `Home`。
- 地图本体放在 `project/user/assets/maps/`，启动时编译或加载为常量表 / 静态结构。
- Flash 不保存地图本体，只保存当前地图编号和当前运行模式。
- 保存入口只有一个：`Home` 页面 `K4` 短按。
- `K4` 长按在所有页面统一返回 `Home`。
- 确认切换地图或模式后只修改 RAM 状态并标记 `Dirty`，不立即写 Flash。

## 菜单树

```text
Home
|-- Run
|   |-- Mode Select
|   `-- Playback
|-- Debug
`-- Info
```

不再单独保留 `Store` 一级菜单。手动 `Load Saved`、`Defaults`、`Save Now` 第一版都不需要，避免菜单入口变多。

## Home 页面

### 显示内容

```text
Home
> Run
  Debug
  Info
Map : V01
Mode: Solve
Save: Saved / Dirty / Error
```

### 按键行为

```text
K1 短按：上一项
K2 短按：下一项
K3 短按：进入当前选中的一级页面
K4 短按：保存当前地图编号和当前运行模式到 Flash
K4 长按：返回 Home，本页无额外动作
```

### 保存语义

`Home/K4` 短按一次性保存：

```text
current_map
run_mode
```

只保存已确认值，不保存候选值：

```text
candidate_map 不写入 Flash
candidate_mode 不写入 Flash
```

如果用户只是移动候选地图或候选模式，但没有按 `K3` 确认，它不属于正式运行选择，`Home/K4` 保存时必须忽略。

保存成功后：

```text
Save: Saved
```

保存失败后：

```text
Save: Error
```

失败时不修改当前 RAM 状态，用户仍可继续运行当前选择。

## Run 页面

`Run` 页面是实际执行入口。用户进入 `Run` 后，可以选择候选地图、进入模式选择并启动算法。

### 显示内容

```text
Run
Map : V01
Pick: V01 / V02 ...
Mode: Browse / Solve / Run / Demo
Save: Saved / Dirty
State: Idle / Pick Map / Running / OK / Fail
```

### 按键行为

```text
K1 短按：上一张候选地图，不立即生效
K2 短按：下一张候选地图，不立即生效
K3 短按：候选地图与当前地图不一致时确认切换；一致时执行当前地图 + 当前模式
K4 短按：返回 Home

K2 长按：进入 Run / Mode Select
K4 长按：返回 Home
```

`Run` 页面不再保留独立选图子页面。地图选择直接在 `Run` 页面完成，但必须按 `K3` 确认后才改变当前地图。

```text
current_map：当前已确认地图，用于执行算法和保存到 Flash
candidate_map：Run 页面 K1/K2 正在选择的候选地图
```

当 `candidate_map != current_map` 时，`K3` 是确认切图键；确认后：

```text
current_map = candidate_map
Save: Dirty
State: Idle
```

当 `candidate_map == current_map` 时，`K3` 才是执行键。不同模式下执行含义如下：

```text
Browse：只刷新并显示当前地图，不执行 BFS
Solve：运行路径算法，成功后进入 Playback 回放页面
Run：第一版等同 Solve；后续在该模式下接真实底盘运动执行
Demo：自动按顺序验证多张地图，记录每张地图 OK / Fail 和耗时
```

离开 `Run` 页面时丢弃未确认的候选地图：

```text
K4 短按返回 Home：candidate_map = current_map
K4 长按返回 Home：candidate_map = current_map
```

未按 `K3` 确认的候选地图不影响保存、不影响执行，也不在下次进入 `Run` 时保留。

`Run` 第一版不直接驱动车体运动。原因是姿态闭环、轮速闭环和地图路径执行还需要独立 HITL 验证；菜单第一阶段只负责选图、选模式、启动算法和显示最短路径回放，避免把算法验证和真实底盘执行耦合在一起。

执行时必须记录本次 BFS 遍历总耗时，并在地图页面显示：

```text
Time: xxx ms
```

第一版只要求显示 BFS 总耗时，不要求显示每个 BFS 状态的内部展开过程。

### BFS 执行方式

第一版在 `candidate_map == current_map` 时，`K3` 触发后同步执行一次 BFS，完成后再更新页面状态：

```text
K3 短按
-> 记录 start_ms
-> 同步调用 solve_map()
-> 记录 elapsed_ms
-> 成功：进入 Playback，默认停在第 0 步
-> 失败：进入 Playback(Fail)，显示初始地图、失败原因和耗时
```

暂不做多帧增量 BFS。原因：

```text
地图固定为 12x16，第一版搜索规模可控
同步执行更容易保证结果、耗时和路径动作串一致
菜单状态机更轻，不需要维护搜索中间态
当前需求重点是验证算法和显示最短路径回放，而不是实时展示 BFS 展开过程
```

如果后续实测 `K3` 后屏幕和按键卡顿明显，再把 `solve_map()` 拆成可轮询的增量搜索接口。

### Demo 执行方式

第一版 `Demo` 默认按顺序验证全部离线地图，不提供 Demo 范围选择页面。

执行流程：

```text
K3 短按
-> 从第 0 张离线地图开始
-> 每次 menu_poll() 最多执行一张地图的 BFS
-> 记录当前地图 OK / Fail 和耗时
-> 下一次轮询再进入下一张
-> 全部完成后停在 Demo 结果页面
```

`Demo` 不能用一个长 `while` 一次跑完全部地图。虽然单张地图 BFS 可以同步执行，但多张地图验证必须拆到菜单状态机里逐张推进，这样 `K4` 停止键才能在两张地图之间被响应。

显示内容：

```text
Demo
Map : 003/020
OK  : 002
Fail: 000
Time: xxx ms
State: Running / Done / Stop
```

按键行为：

```text
K4 短按：停止 Demo 并返回 Run
K4 长按：停止 Demo 并返回 Home
```

第一版不保存 Demo 结果到 Flash，只在当前上电周期显示。原因是 Demo 用于算法验证，不是运行配置；Flash 只保存当前地图编号和当前运行模式。

## Run / Mode Select 页面

该页面用于选择运行模式。

`Mode Select` 页面使用候选模式。`K1` / `K2` 只移动候选项，不立即修改当前运行模式；只有 `K3` 确认后才生效。`K4` 取消时丢弃候选项，返回 `Run`。

```text
run_mode：当前已确认模式，用于执行算法和保存到 Flash
candidate_mode：Mode Select 页面 K1/K2 正在选择的候选模式
```

### 显示内容

```text
Mode Select
> Browse
  Solve
  Run
  Demo
```

### 按键行为

```text
K1 短按：上一候选模式
K2 短按：下一候选模式
K3 短按：确认当前模式并返回 Run
K4 短按：取消并返回 Run
K4 长按：返回 Home
```

确认模式后：

```text
Save: Dirty
```

取消或离开 `Mode Select` 时丢弃未确认的候选模式：

```text
K4 短按返回 Run：candidate_mode = run_mode
K4 长按返回 Home：candidate_mode = run_mode
```

## Run / Playback 页面

`Playback` 页面用于在地图上显示最短路径如何完成当前地图。进入条件是 `Run` 页面 `K3` 执行 BFS 成功。

### 显示内容

```text
Playback
Map : V01
Step: 000/086
Act : U / D / L / R / u / d / l / r
Time: xxx ms
State: Paused / Playing / Done / Fail
```

地图区域应动态显示：

```text
小车当前位置
箱子当前位置
目标点
墙和空地
```

第一版只显示小车所在格，不显示小车朝向。朝向属于后续真实底盘执行层，不作为 BFS 最短路径回放的验收内容。

Playback 第一版使用色块地图，不使用字符地图作为主显示。颜色参考逐飞智能视觉组文章中的虚拟地图语义：

```text
墙体：灰色
箱子：黄色
目标点：紫色
炸弹：红色，第一版可预留但不参与基础 Push Box 验证
小车：绿青色
空地：黑色或白色背景，按屏幕可读性选择
```

XSB 字符只作为地图源数据和调试输出，不作为 Playback 主视觉。

### 进入流程

```text
K3 短按
-> 对当前地图跑 BFS
-> 记录 BFS 总耗时
-> 如果失败：显示 Fail、原因和耗时
-> 如果成功：进入 Playback
-> 默认暂停在第 0 步
```

失败时也进入 `Playback`，但状态为 `Fail`，地图停在初始状态。这样成功和失败共用同一页面，不额外增加失败弹窗。

### 按键行为

```text
K1 短按：上一步
K2 短按：下一步
K3 短按：播放 / 暂停
K4 短按：返回 Run
K4 长按：返回 Home
```

### 播放规则

```text
默认状态：Paused，停在第 0 步
K3 后：Playing，每 300ms 自动推进一步
播放结束：Done，停在最后一步
手动 K1/K2 单步时：保持 Paused
```

回放使用 solver 已生成的最短路径动作串，不重新运行 BFS。

第一版固定播放速度：

```text
PLAYBACK_STEP_MS = 300
```

播放速度不进入菜单，也不写入 Flash。后续只有在实测需要时再增加运行时配置。

### 绘制策略

第一版 `Playback` 每推进一步都整张重画 12x16 色块地图区域，不做局部脏矩形更新。

```text
从初始地图 + actions[0..step] 推导当前地图状态
重绘 12x16 色块地图区域
更新 Step / Act / Time / State 行
```

选择整张重画的原因：

```text
12x16 只有 192 格，刷新成本可控
实现简单，不容易留下残影
不需要维护上一帧脏区域
小车、箱子、目标点叠加关系更容易保证正确
```

后续只有在实测刷新明显卡顿时，才考虑改为局部更新。

### 色块绘制约束

第一版保持轻量，不引入图片资源和额外图形库。

IPS200 驱动尺寸边界：

```text
IPS200_PORTAIT / IPS200_PORTAIT_180：240 x 320
IPS200_CROSSWISE / IPS200_CROSSWISE_180：320 x 240
```

当前 `app.c` 使用 `IPS200_PORTAIT`，第一版菜单按竖屏 `240 x 320` 设计，不切横屏。

推荐布局：

```text
SCREEN_W = 240
SCREEN_H = 320
MAP_COLS = 16
MAP_ROWS = 12
CELL_SIZE = 14
MAP_W = 16 * 14 = 224
MAP_H = 12 * 14 = 168
MAP_X = 8
MAP_Y = 112
```

竖屏区域安排：

```text
0-15    页面标题
16-111  页面状态文字
112-279 12x16 色块地图
288-303 按键提示第 1 行
304-319 按键提示第 2 行
```

屏幕底部保留两行英文短按键提示。提示必须短，不超过 30 个 8x16 字符，避免越界或挤压地图。地图区域结束于 y=279，按键提示从 y=288 开始，中间留 8px 空白。

不使用 `CELL_SIZE = 15` 的原因是地图宽度刚好 240px，没有边距，后续画边框或网格线容易越界；`14px` 留出左右各 8px，显示更稳。

### 统一时间基准

菜单、BFS 耗时统计和 Playback 自动步进共用同一个毫秒时间基准，不再新增 PIT 中断。

推荐基于逐飞库 `libraries/zf_driver/zf_driver_timer.h` 中的 GPT 定时器实现：

```text
timer_init(GPT_TIM_1, TIMER_MS)
timer_start(GPT_TIM_1)
timer_get(GPT_TIM_1)
```

对上层只暴露两个接口：

```c
void timebase_init(void);
uint32 time_ms(void);
```

使用方式：

```text
BFS 开始前记录 start_ms = time_ms()
BFS 结束后 elapsed_ms = time_ms() - start_ms
Playback 中用 time_ms() - last_step_ms >= PLAYBACK_STEP_MS 判断是否推进下一步
```

选择 GPT 而不是 PIT 的原因：

```text
PIT_CH0 已用于 5ms IMU/yaw 更新
PIT_CH1 已用于 20ms 姿态控制和车轮速度闭环
PIT_CH3 在逐飞 TSL1401 头文件中作为采集定时器预留
菜单计时只需要读当前时间，不需要额外中断
```

`time_ms()` 必须是非阻塞读数函数，不允许内部调用 `system_delay_ms()` 或通过等待循环制造时间间隔。

推荐实现方式：

```text
每个地图格子用固定像素方块绘制
用 ips200_draw_line() 多行填充矩形，或封装一个小的 fill_rect()
不为 12x16 地图创建整屏 framebuffer
不使用动态内存
地图色块区域和文字状态区域分开，避免每步重绘整屏
```

推荐颜色映射：

```text
SCREEN_TEXT_COLOR = RGB565_BLACK
SCREEN_BG_COLOR   = RGB565_WHITE
WALL_COLOR    = RGB565_BLACK
BOX_COLOR     = RGB565_YELLOW
TARGET_COLOR  = RGB565_PURPLE
CAR_COLOR     = RGB565_CYAN 或 RGB565_66CCFF
BOMB_COLOR    = RGB565_RED
EMPTY_COLOR   = RGB565_WHITE
GRID_COLOR    = RGB565_GRAY
```

绘制叠加优先级：

```text
墙 > 小车在目标上 > 箱子在目标上 > 小车 > 箱子 > 目标 > 空地
```

其中“小车在目标上”和“箱子在目标上”可以用同一色块加边框区分；第一版如果边框实现成本高，可以优先使用小车 / 箱子的实体颜色，目标状态由算法状态行辅助确认。

## Flash 保存内容

第一版只保存少量运行选择：

```text
magic
version
checksum
current_map
run_mode
```

`checksum` 是轻量完整性检查，不使用复杂 CRC32。第一版可以对除 `checksum` 外的记录字段按字节做 `uint16` 累加校验，用于识别空页、半写和明显损坏记录。

不保存：

```text
地图本体
BFS 结果
路径动作串
临时页面状态
当前光标位置
```

## Flash 存储策略

基于 `libraries/zf_driver/zf_driver_flash.h` 提供的接口实现：

```text
flash_init()
flash_read_page()
flash_write_page()
flash_erase_page()
```

逐飞 Flash 驱动粒度：

```text
FLASH_BASE_ADDR   = 0x70000000
FLASH_PAGE_SIZE   = 4KB
FLASH_PAGE_NUM    = 8
FLASH_SECTOR_SIZE = 32KB
sector_num 范围   = 0..127
page_num 范围     = FLASH_PAGE_0..FLASH_PAGE_7
FLASH_DATA_BUFFER_SIZE = 1024 个 uint32 单元
```

`flash_write_page()` 内部会先调用 `flash_check()` 检查目标页是否为空；如果目标页非空，会自动调用 `flash_erase_page()` 清除该页后再写入。因此业务层保存配置时不需要额外手动擦页，直接写目标页即可。

接口使用约束：

```text
flash_read() 宏的 offset 单位是 4 字节，不是字节偏移
flash_read_page() / flash_write_page() 的 len 单位是 uint32 个数，不是字节数
flash_check() 返回 0 表示空页，返回非 0 表示该页已有数据
flash_erase_page() 返回 0 表示成功，非 0 表示失败
flash_write_page() 返回 0 表示成功，非 0 表示失败
```

第一版 `settings.c` 不使用 `flash_union_buffer`、`flash_read_page_to_buffer()` 或 `flash_write_page_from_buffer()`，只使用局部 `uint32` 缓冲区配合 `flash_read_page()` / `flash_write_page()`。原因是配置记录很小，局部缓冲更直观；同时当前 `flash_write_page_from_buffer()` 没有把内部 `flash_write_page()` 的返回值传出来，不适合作为保存成功判断。

推荐使用最后一个独立保留扇区：

```text
sector 127
page 0 固定保存配置
```

写入规则：

```text
启动：读取 page0
校验：magic、version、checksum 均有效才接受
保存：Home/K4 每次都写入 page0
失败：保持 RAM 配置不变，Info 页面显示错误
```

保存流程：

```text
准备 settings_record
计算 checksum
调用 flash_write_page(sector 127, page 0)
写后重新 flash_read_page() 校验 magic、version、checksum 和字段值
校验成功才更新 RAM 中的 Saved 状态
```

第一版不做“内容未变化则跳过写入”的优化。`Home/K4` 的语义保持直接：用户按保存，就执行一次 Flash 保存流程；保存成功显示 `Saved`，失败显示 `Error`。

注意：配置页使用最后一个 32KB 扇区。`project/mdk/scf/MIMXRT1064xxxxx_flexspi_nor.scf` 需要通过缩小程序 Flash 区域保留 sector 127，避免程序镜像和配置页共用同一地址范围。当前实现已将 `m_text_size` 缩小到 `0x003F5C00`，程序镜像结束地址为 `0x703F8000`。

## 推荐模块拆分

```text
project/user/inc/settings.h
project/user/src/settings.c
project/user/inc/timebase.h
project/user/src/timebase.c
project/user/inc/screen.h
project/user/src/screen.c
project/user/inc/menu.h
project/user/src/menu.c
project/user/inc/app.h
project/user/src/app.c
```

`settings.c/h` 负责：

```text
默认配置
Flash 初始化
Flash 读取
Flash 保存
checksum 校验
Saved / Dirty / Error 状态
```

`timebase.c/h` 负责：

```text
统一毫秒时间基准
timebase_init()
time_ms()
基于 GPT_TIM_1 + TIMER_MS
```

`screen.c/h` 负责：

```text
IPS200 初始化
文本行绘制
12x16 色块地图绘制
Playback 地图状态绘制
不处理按键
不运行 BFS
不读写 Flash
```

`menu.c/h` 负责：

```text
页面状态机
按键事件分发
Home 静态表和页面跳转
current_map / candidate_map
run_mode / candidate_mode
调用 settings / screen / solver / plan_output
```

`app.c` 只保留应用级 glue：

```text
app_init()
app_poll()
```

`app.c` 负责初始化和轮询各子模块：

```text
timebase_init()
settings_init()
screen_init()
menu_init()
menu_poll()
```

后续把当前 `app.c` 里的 `ui_*` 迁移到 `screen.c`，`handle_keys()` 和执行入口迁移到 `menu.c`，保持 `main.c` 继续轻量。

第一版不额外拆 `key_event.c`、`playback.c`、`demo.c`。如果后续 `menu.c` 明显膨胀，再按页面职责拆分。

### 最小接口合同

第一版只暴露必要接口，避免模块之间互相读写内部状态。

```c
/* settings.h */
void settings_init(void);
uint8 settings_get_map(void);
run_mode_enum settings_get_mode(void);
save_state_enum settings_get_save_state(void);

uint8 settings_save(void);
```

```c
/* timebase.h */

uint32 time_ms(void);
```

```c
/* screen.h */
void screen_init(void);
void screen_draw_home(...);
void screen_draw_run(...);
void screen_draw_mode_select(...);
void screen_draw_playback(...);
void screen_draw_info(...);
```

```c
/* menu.h */
void menu_init(void);

```

初始化默认值：

```text

candidate_map = current_map
candidate_mode = run_mode

```

`settings.c` 只保存已确认的 `current_map` 和 `run_mode`，不保存 `candidate_map` 或 `candidate_mode`。`menu.c` 是候选态的唯一拥有者。

## 菜单实现方式

第一版不使用链表、动态菜单树或堆内存。菜单页面数量虽然后续会增加，但仍属于编译期已知页面，适合用静态表扩展。

推荐结构：

```text
页面跳转：menu_page_enum + switch
Home 一级菜单：const home_item_struct 静态表
简单选项列表：const char *const 静态表
页面行为：每页一个 handle() 和 draw()
```

Home 一级菜单不要在绘图代码里写死。推荐用静态表描述：

```c
typedef struct
{
    const char *name;
    menu_page_enum page;
} home_item_struct;

static const home_item_struct home_items[] =
{
    {"Run",  MENU_PAGE_RUN},
    {"Debug", MENU_PAGE_DEBUG},
    {"Info", MENU_PAGE_INFO},
};
```

后续新增 Home 子页面时，只需要：

```text
1. 在 menu_page_enum 里增加页面
2. 在 home_items[] 里增加入口
3. 增加该页面的 handle() 和 draw()
4. 在 page switch 里接入该页面
```

不使用链表的原因：

```text
页面是编译期固定资源，不需要运行时插入或删除
静态表更容易看清 Home 下有哪些入口
无动态内存，掉电保存和菜单状态更容易验证
指针链路少，Keil 调试时状态更直观
```

不建议第一版把所有页面都做成复杂函数指针框架。Home 入口可以表驱动，但页面内部逻辑仍使用明确的 `switch` 和独立函数，避免为了扩展性牺牲可读性。

## 按键事件适配

菜单页面不直接处理逐飞驱动的 `key_get_state()` 原始状态，而是在 `menu.c` 内部封装一次性按键事件。

逐飞 `zf_device_key` 当前行为：

```text
KEY_SHORT_PRESS：按键释放后产生
KEY_LONG_PRESS：按住超过 KEY_LONG_PRESS_PERIOD 后持续保持
KEY_LONG_PRESS_PERIOD = 1000ms
```

如果页面直接读取 `KEY_LONG_PRESS`，长按期间可能被重复消费；如果长按释放后又产生短按语义，还会造成 `K4 长按返回 Home` 和 `Home/K4 短按保存` 互相干扰。

推荐在 `menu.c` 内部定义：

```c
typedef enum
{
    KEY_EVENT_NONE,
    KEY_EVENT_K1_SHORT,
    KEY_EVENT_K2_SHORT,
    KEY_EVENT_K3_SHORT,
    KEY_EVENT_K4_SHORT,
    KEY_EVENT_K1_LONG,
    KEY_EVENT_K2_LONG,
    KEY_EVENT_K3_LONG,
    KEY_EVENT_K4_LONG,
} key_event_enum;

static key_event_enum key_read_event(void);
```

事件规则：

```text
短按：松手后触发一次
长按：超过 1000ms 后触发一次
长按触发后，直到该按键释放前不再重复触发
长按释放时不再补发短按
菜单页面只处理 key_event_enum，不直接处理 key_get_state()
```

第一版不单独拆 `key_event.c/h`。原因是目前只有菜单需要事件化按键；如果后续其他模块也需要同一套短按 / 长按事件，再把它从 `menu.c` 拆成独立模块。

## 参考 `menu.c` 的取舍

可以参考 `wushua-----/project/service/menu.c` 的菜单思路，但不照搬其完整实现。

可借鉴的部分：

```text
统一页面状态变量
每页保留当前光标位置
K1/K2 作为上一项 / 下一项
K3 作为进入 / 确认
K4 作为返回
Home 页面 K4 作为保存动作
保存后短暂显示 Saved 提示
```

不照搬的部分：

```text
不使用十进制页面编码，例如 11、12、21、22
不做 1000 行级参数调试菜单
不引入通用数值编辑器和步进倍率
不使用阻塞式菜单独占前台任务
不维护复杂的多页参数表
不把保存动作散落到多个页面
```

本项目第一版只需要 6 个页面状态：

```text
MENU_PAGE_HOME
MENU_PAGE_RUN
MENU_PAGE_MODE_SELECT
MENU_PAGE_PLAYBACK
MENU_PAGE_DEBUG
MENU_PAGE_INFO
```

推荐使用简单 `enum` 和 `switch` 实现，不引入动态菜单树、函数指针表或堆内存。

推荐运行方式：

```text
app_poll()
  menu_poll()
    key_scanner()
    key_read_event()
    更新页面状态
    标记是否需要重绘
  menu_draw_if_needed()
```

菜单必须是非阻塞状态机。算法执行可以在 `K3` 触发时同步运行，但菜单本身不应通过长时间 `while` 循环接管主循环。
