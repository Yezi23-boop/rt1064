# project/user 用户代码可读性优化计划

## 目标

对 `project/user` 下的用户代码做一轮可读性优化，让代码满足：

- 小白能顺着文件名、函数名和注释看懂主要流程。
- 代码直观、简单，减少不必要的嵌套。
- 注释解释原因、约束、风险和单位，不逐行翻译代码。
- 不改变当前菜单交互、算法结果、控制数学和硬件行为。

本计划是给后续 agent 执行的工程合同。执行时优先保持行为稳定，不追求“大重构”。

## 优化范围

只允许修改：

```text
D:/rt1064/RT1064_Library/SeekFree_RT1064_Opensource_Library/project/user
```

允许涉及：

- `project/user/inc`
- `project/user/src`
- `project/user` 下已有资源文件

除非文件名变更导致工程无法编译，否则不要主动修改 `project/mdk`。如果必须同步 Keil 工程文件，只做路径同步，不做工程配置优化。

## 禁止范围

不允许修改：

- `libraries`
- 逐飞驱动
- 底层板级支持代码
- OpenART / OpenMV 脚本
- 仓库文档，除非用户另行要求

不允许改变：

- 菜单页面结构
- 按键行为
- BFS 核心流程
- BFS 求解结果
- 麦轮混控公式
- 姿态环、PID、控制参数
- 初始化顺序
- ISR 接入关系
- 5ms IMU 更新节拍
- 20ms 控制更新节拍
- Flash 保存语义
- SDRAM / 片上 RAM 的既定放置原则

## 允许修改

允许在 `project/user` 内做：

- 文件名优化
- 函数名优化
- 公共接口名优化
- 类型名、宏名、变量名优化
- 局部函数拆分
- 早返回改写
- 删除无意义嵌套
- 删除重复代码
- 补充或重写注释
- 删除“逐行翻译代码”的注释

注意：本轮不拆文件。即使文件偏长，也先在现有文件内整理函数结构和命名。

## 注释规则

按 `embedded-c-cpp-comment-style` 执行。

### `.h` 文件

`project/user/inc` 中的公开接口要补 Doxygen 契约注释：

- 公开函数
- 公开类型
- 公开枚举
- 公开宏
- 重要结构体字段

函数注释至少说明：

- 函数用途
- 参数含义、单位、合法范围
- 返回值语义
- 重要副作用
- 是否依赖 ISR、定时周期、Flash、SDRAM、硬件初始化或不可阻塞上下文

### `.c` 文件

`.c` 中优先使用 `//` 注释，只解释代码字面看不出的信息：

- 为什么这样写
- 受什么硬件、时序、协议、并发或内存约束
- 改掉会有什么风险
- 单位是什么
- 谁写谁读
- 哪个上下文调用

不要写这类注释：

```c
// 设置变量
// 调用函数
// i 加一
// 进入循环
```

应该写这类注释：

```c
// 只在切换页面时清屏，避免每次刷新整屏导致闪烁。
// 该 buffer 只用于显示填充，不参与 BFS，放 cacheable SDRAM 可以减少片上 RAM 压力。
// 5ms 节拍只更新 IMU 姿态，20ms 节拍再做控制闭环，避免控制周期内重复读传感器。
```

## 执行步骤

3. 执行一次 Keil 全量编译，记录基线：


成功标准：


### 第 2 步：补 `project/user/inc` 公开接口注释

按文件逐个补 Doxygen：

- `app.h`
- `base_io.h`
- `drive_config.h`
- `drive_control.h`
- `isr.h`
- `map_types.h`
- `maps.h`

- `motion_math.h`
- `plan_output.h`
- `screen.h`
- `settings.h`
- `solver.h`

要求：

- 公开函数必须说明用途、参数、返回值、副作用。
- enum 必须说明每个值的业务含义。
- 宏必须说明单位、范围或来源。
- 涉及 ISR、Flash、SDRAM、控制周期的接口必须说明调用上下文。

禁止：

- 不要给每个简单 include 写注释。
- 不要用注释重复函数名。

### 第 3 步：优化命名

优先优化小白不容易理解的名字：

- 太泛的名字：`process`、`handle`、`state`、`flag`。
- 不能看出单位的名字：时间、计数、角度、速度。
- 不能看出上下文的名字：ISR 变量、Flash 状态、页面状态。

命名建议：

- 时间变量带单位：`elapsed_ms`、`control_dt_s`。
- 页面状态带含义：`current_page`、`playback_state`。
- Flash 保存状态说明用途：`save_state`、`settings_dirty`。
- 控制量保留领域词：`vx`、`vy`、`vz` 可以保留，但注释必须说明车体坐标含义。

改名后必须搜索旧名：

```powershell
Get-ChildItem -Path project/user -Recurse -File | Select-String -Pattern "旧名字"
```

成功标准：

- 旧名字不再作为代码标识符出现。
- 如果旧名字只在计划文档或历史说明中出现，要确认不是当前源码入口。

### 第 4 步：减少不必要嵌套

优先处理：

- `menu.c`
- `screen.c`
- `settings.c`
- `drive_control.c`
- `motion_math.c`

推荐方式：

- 用早返回减少 `else`。
- 把重复的三到五行逻辑抽成小函数。
- 状态机分支保持一屏内能读完。
- 一个函数只表达一层逻辑。

示例：

```c
if(error)
{
    return;
}

do_normal_work();
```

不推荐：

```c
if(!error)
{
    if(ready)
    {
        if(enabled)
        {
            do_normal_work();
        }
    }
}
```

### 第 5 步：保守处理 `solver.c`

`solver.c` 只允许：

- 改命名
- 补注释
- 提取不会改变 BFS 流程的小辅助函数

不允许：

- 重排 BFS 核心流程
- 改状态编码方式
- 改队列策略
- 改父节点回溯策略
- 改地图拆解策略
- 改最短路径选择规则

重点注释：

- 队列含义
- visited / parent / action 数组含义
- 为什么 BFS 可以保证单次最短
- 多箱任务如何拆成单箱子问题
- 动作字符大小写含义
- `|` 分隔符的意义

### 第 6 步：保守处理底盘控制

底盘相关文件只允许：

- 改命名
- 补注释
- 减少嵌套

不允许：

- 改 PID 参数
- 改姿态 PD 参数
- 改麦轮混控公式
- 改 `vx/vy/vz` 语义
- 改 5ms / 20ms 周期
- 改轮序映射
- 改电机方向符号
- 改编码器方向符号

重点注释：

- `vx`：车体右方向速度分量。
- `vy`：车体前方向速度分量。
- `vz`：车体自转速度分量。
- 姿态环使用 PD，不使用积分。
- `wheel_norm[-1,+1]` 是归一化中间量，不是编码器目标。
- `MAX_WHEEL_TARGET_COUNT` 是归一化满幅对应的 20ms 编码器目标计数。

### 第 7 步：检查行为边界

完成代码整理后检查：

- 菜单页面名称没有变化。
- K1/K2/K3/K4 行为没有变化。
- K4 长按仍是返回 Home。

- Mode Select 仍按原规则切换和确认。
- Playback 仍按原规则前进、后退、播放、返回。

## 每类文件的具体要求

### `main.c`

- 保持轻量。
- 只允许补启动顺序说明和少量命名优化。
- 不允许把菜单、求解、控制逻辑塞回 `main.c`。

### `isr.c`

- 明确 5ms 和 20ms 的职责边界。
- 不改中断调用顺序。
- 注释要说明 ISR 内不能做阻塞操作。

### `menu.c`

- 页面状态机必须直观。
- 每个页面的按键处理函数只处理当前页面。
- K4 长按返回 Home 的全局逻辑保持统一。
- 运行模式确认逻辑可以拆小函数，但不能改交互。

### `screen.c`

- 屏幕绘制函数只负责显示，不决定业务状态。
- 色块 buffer 继续放 cacheable SDRAM。
- 注释说明为什么避免每次整屏清屏。
- 注释说明地图色块颜色语义。

### `settings.c`

- Flash 保存只保存地图编号和运行模式。
- 注释说明 Flash 写入代价和为什么不在每次切换时写。
- 不引入 sequence、CRC 或复杂保护，除非用户重新要求。

### `solver.c`

- 保持 BFS 流程。
- 注释解释队列、父节点、动作回溯和任务拆解。
- 不做性能优化，除非编译或实测暴露问题。

### `drive_control.c`

- 注释说明 5ms/20ms 两级节拍。
- 不改控制节拍。
- 不改目标速度到车轮目标的调用链。

### `motion_math.c`

- 注释解释车体坐标系。
- 注释解释姿态 PD 不使用积分的原因。
- 注释解释归一化的目的和风险。
- 不改公式结果。

### `base_io.c`

- 注释说明 IMU660RC/660RA 同接口同协议的使用前提。
- 注释说明编码器计数方向和 PWM 方向由配置表校正。
- 不改硬件映射。

### `wheel_pid.c`

- 注释说明增量式 PID 的输入输出单位。
- 不改 PID 参数。
- 不改 PWM 限幅逻辑。
